## 도커의 구조

### 도커 위치 확인 명령어

which docker

/usr/bin/docker

명령어의 파일이 위치한 경로를 출력

### 실행 중인 도커 프로세스 확인

ps aux | grep docker

실행 중인 프로세스 목록 출력

컨테이너, 이미지를 다루는 명령어는 /usr/bin/docker 에서 실행되지만

도커 엔진의 프로세스는 /usr/bin/dockerd 파일로 실행됩니다.

docker 명령어가 실제 도커 엔진이 아닌 클라이언트로서의 도커이기 때문입니다.

도커의 구조 = 클라이언트로서의 도커 + 서버로서의 도커

### 도커 서버

실제로 컨테이너 생성, 실행하며 이미지를 관리하는 주체 ⇒ dockerd 프로세스로 동작

도커 엔진은  외부에서 API 입력을 받아 도커 엔진의 기능을 수행

도커 프로세스가 실행되어 서버로서 입력을 받을 준비가 된 상태를 도커 데몬이라고 한다.

### 도커 클라이언트

도커 데몬은 API 입력을 받아 도커 엔진의 기능을 수행하는데 이 API를 사용할 수 있도록 CLI를 제공하는 것

사용자가 docker로 시작하는 명령어를 입력하면 도커 클라이언트를 사용하는 것

도커 클라이언트는 입력된 명령어를 로컬에 존재하는 도커 데몬에게 API로서 전달

이때 도커 클라이언트는 /var/run/docker.sock에 위치한 유닉스 소켓을 통해 도커 데몬의 API를 호출

도커 클라이언트가 사용하는 유닉스 소켓은 같은 호스트 내에 있는 도커 데몬에게 명령을 전달할 때 사용

tcp로 원격에 있는 도커 데몬을 제어하는 방법도 있음

![Untitled](%E1%84%83%E1%85%A9%E1%84%8F%E1%85%A5%20%E1%84%83%E1%85%A6%E1%84%86%E1%85%A9%E1%86%AB%20b108b1d0ee644f7c86a582e57d7e3328/Untitled.png)

터미널, putty 등으로 도커가 접속된 호스트에 접속해 docker 명령어를 입력하면 하위 과정으로 도커가 제어됨

1. 사용자가 docker ps 같은 도커 명령어를 입력
2. /usr/bin/docker 는 /var/run/docker.sock 유닉스 소켓을 사용해 도커 데몬에게 명령어를 전달
3. 도커 데몬은 이 명령어를 파싱하고 명령어에 해당하는 작업을 수행
4. 수행 결과를 도커 클라이언트에게 반환하고 사용자에게 결과를 출력

아무런 설정을 하지 않았을 때 일반적으로 도커 데몬을 제어하는 순서임. 옵션 추가 시 과정이 추가될 수 있음

## 도커 데몬 실행

도커 데몬은 일반적으로 아래와 같은 명령어로 시작, 정지할 수 있음

우분투에서는 도커가 설치되면 자동적으로 서비스로 등록되므로 호스트가 재시작되더라도 자동으로 실행

service docker start

service docker stop

레드햇 계열의 운영체제는 도커를 설치해도 자동으로 실행되도록 설정되지는 않음

도커를 자동적으로 실행하도록 설정하려면 아래의 명령어로 docker 서비스를 활성화해야 함

systemctl enable docker 

도커 서비스는 dockerd로 도커 데몬을 실행함

서비스를 사용하지 않고 도커 데몬을 실행할 수도 있음

dockerd 명령어 또한 /usr/bin/dockerd로서 존재하기 때문에 docker 명령어와 같이 바로 사용할 수 있음

service docker stop

dockerd

dockerd를 입력하면 도커 데몬이 실행됨.

도커 데몬에 대한 각정 정보가 출력되는대, 마지막 /var/run/docker.sock에서 입력(listen)을 받을 수 있는 상태라는 메세지가 출력됨

터미널을 하나 더 연 다음 도커 명령어를 입력하면 이전처럼 도커 사용 가능함

터미널에 실행된 도커 데몬을 종료하려면 ctrl + c를 입력하면 됨

```jsx
TIP
디버깅이나 도커 자체의 트러블 슈팅이 필요하다면 도커 데몬을 직접 실행하는 것이 나을 수 있음
그러나 도커 데몬을 실행하면 하나의 터미널을 차지하는 포그라운드 상태로 실행되기 때문에 운영 및 관리 측면에서 부적절
실제 운영 환경에서는 도커 데몬을 직접 실행하기보다 service, systemctl 명령어를 통해 리눅스 서비스로서 관리하는 것이 좋음
```

## 도커 데몬 설정

도커 데몬에 적용할 수 있는 옵션 확인

dockerd —help

```jsx
--insecure-registry : 레즈스토리 컨테이너 구축 때 사용

```

### 도커 데몬 제어: -H

-H 옵션은 도커 데몬의 API를 사용할 수 있는 방법을 추가

아무런 옵션 설정 없이 도커 데몬을 실행하면 도커 클라이언트인 /usr/bin/docker를 위한 유닉스 소켓인 /var/run/docker.sock을 사용

dockerd를 입력해 도커 데몬을 실행해도 도커 클라이언트의 CLI를 사용할 수 있습니다.

dockerd

dockerd -H unix:///var/run/docker.sock

위 두 명령어는 차이가 없습니다.

-H에 IP 주소와 포트 번호를 입력하면 원격 API인 Docker Remote API로 도커를 제어할 수 있음

Remote API는 도커 클라이언트와 다르게 로컬에 있는 도커 데몬이 아니더라도 제어할 수 있으며,

RESTful API 형식을 띠고 있으므로 HTTP 요청으로 도커를 제어할 수 있음

도커 데몬을 실행하면 호스트에 존재하는 모든 네트워크 인터페이스의 IP주소와 2375 포트를 바인딩해 입력 받음

dockerd -H tcp://0.0.0.0:2375

-H에 unix:///var/run/docker.sock을 지정하지 않고, 위와 같이 Remote API만을 위한 바인딩 주소를 입력했다면

유닉스 소켓은 비활성화되므로 도커 클라이언트를 사용할 수 없게 되며, docker로 시작하는 명령어를 사용할 수 없음

일반적으로 도커 클라이언트를 위한 유닉스 소켓과 Remote API를 위한 바인딩 주소를 동시에 설정

호스트 모든 네트워크 인터페이스 카드에 할당된 IP주소와 2375 포트로 도커 데몬을 제어함과 동시에 도커 클라이언트도 사용 가능 예시

dockerd -H unix:///var/run/docker.sock -H tcp://0.0.0.0:2375
